# SPEC.md — Clearline MVP Spec

## 1. Product Summary

Clearline is a USDC Payment Inbox for Solana wallets.

It helps people and teams understand incoming USDC transfers by classifying each payment into a clear operational state.

Clearline tells users what each incoming payment is without forcing them to open explorers, manually inspect transactions, or maintain spreadsheets.

Core idea:

> USDC can move globally in seconds, but wallet payments arrive without context.  
> Clearline tells you what each transfer is.

---

## 2. Product Type

Clearline is:

- a payment inbox
- a payment classification system
- an exception handling tool for wallet-based USDC transfers

Clearline is not:

- an invoicing tool
- a billing platform
- an accounting system
- a financial dashboard
- a treasury product
- a fiat payment processor

---

## 3. Primary User

Primary users are people or teams receiving repeated USDC payments directly to a Solana wallet from multiple senders.

Examples:

- freelancers
- agencies
- founders
- crypto-native teams
- grant recipients
- service providers

The defining trait is not geography.

The defining trait is:

- multiple incoming USDC transfers
- multiple senders
- direct wallet payments
- need to understand what each payment represents

---

## 4. Core Problem

Wallet-based USDC payments are fast, global, and cheap, but they often arrive without enough business context.

Common problems:

- sender is unknown
- payment amount is wrong
- payment is partial
- payment is duplicated
- payment has no reference
- payment must be manually checked in a blockchain explorer
- payment status must be tracked manually in a spreadsheet

This creates operational confusion.

---

## 5. Core Solution

Clearline creates a structured layer around incoming USDC transfers.

The system allows users to create expected payments, generate Solana Pay links with references, monitor incoming USDC transfers, classify each payment, and surface exceptions in a Payment Inbox.

Clearline focuses on answering one question:

> What is this incoming USDC payment?

---

## 6. MVP Scope

The MVP must support:

- creating expected payments
- generating Solana Pay links with references
- monitoring incoming USDC transfers
- detecting real transactions
- extracting references when available
- falling back to RPC when needed
- classifying payments
- displaying exceptions
- manually resolving unknown payments
- showing proof pages

The MVP does not need:

- user accounts
- multi-wallet support
- fiat payments
- subscriptions
- recurring billing
- AI features
- charts
- analytics
- production-grade accounting logic
- mainnet deployment

---

## 7. Core User Flows

### 7.1 Create Expected Payment

User creates an expected payment with:

- label
- amount in USDC
- optional due date

System generates:

- unique Solana reference public key
- Solana Pay URL
- expected payment record in database

Expected result:

- expected payment appears in the system
- Solana Pay link can be used to pay
- reference is stored for matching

---

### 7.2 Pay With Solana Pay Reference

Payer sends USDC using a Solana Pay link generated by Clearline.

The transaction should include the generated reference public key.

Expected result:

- payment lands on Solana Devnet
- Clearline can identify the reference
- Clearline can match the payment to the expected payment

---

### 7.3 Detect Incoming USDC Transfer

Helius webhook sends transaction data to Clearline.

System checks:

- authorization header
- transaction signature
- USDC mint
- recipient wallet or recipient USDC ATA
- amount
- raw payload

Expected result:

- valid incoming USDC transfer is stored
- duplicate webhook delivery is ignored
- raw payload is saved before classification

If a webhook payload contains no valid incoming USDC transfer, Clearline returns success and does not create a transaction record.

---

### 7.4 Extract Reference

System attempts to extract a reference public key from the transaction.

Primary method:

- inspect Helius enhanced payload account keys

Fallback method:

- fetch transaction by signature via RPC
- inspect account keys
- compare against stored expected payment references

Expected result:

- matching reference is found when present
- transaction becomes unknown if no reference is found

---

### 7.5 Classify Exact Payment

If incoming payment matches an expected payment reference and amount is exact:

Status:

- `paid`

Expected reason:

- exact amount received
- reference matched

---

### 7.6 Classify Partial Payment

If incoming payment matches an expected payment reference and amount is lower than expected:

Status:

- `partial`

Expected reason:

- expected amount
- received amount
- reference matched

---

### 7.7 Classify Overpaid Payment

If incoming payment matches an expected payment reference and amount is higher than expected:

Status:

- `overpaid`

Expected reason:

- expected amount
- received amount
- reference matched

---

### 7.8 Classify Duplicate Payment

If an expected payment is already marked as paid and another transaction arrives with the same matched reference:

Status:

- `duplicate`

Expected reason:

- expected payment already marked paid

The expected payment status should not be changed.

---

### 7.9 Classify Unknown Payment

If incoming USDC transfer has no matching reference:

Status:

- `unknown`

Expected reason:

- no matching Solana Pay reference found

Unknown payments must appear in the Payment Inbox under Needs Attention.

---

### 7.10 Manual Resolution

User manually assigns an unknown payment to an expected payment.

System:

- links transaction to selected expected payment
- re-runs the same classifier used by webhook ingestion
- updates transaction status
- updates expected payment status if applicable

Expected result:

- unknown payment becomes paid, partial, overpaid, or duplicate depending on the selected expected payment and amount
- classification reason updates
- no heuristic matching is required

If an unknown transaction is manually assigned to an expected payment that is already `paid`, the transaction becomes `duplicate` and the expected payment remains `paid`.

---

### 7.11 View Proof Page

User opens proof page for a transaction.

Proof page shows:

- transaction signature
- amount
- sender wallet
- recipient wallet
- status
- classification reason
- linked expected payment if any
- Solana Explorer link

Expected result:

- user can verify what happened onchain
- user can understand why Clearline classified the payment that way

---

## 8. System Behaviors

### 8.1 Reference Match Behavior

If a transaction contains a reference matching an expected payment:

- match transaction to expected payment
- classify based on amount
- update transaction status
- update expected payment status if applicable

---

### 8.2 No-Reference Behavior

If a transaction has no matching reference:

- classify as `unknown`
- do not guess
- do not match by amount
- do not match by sender wallet
- surface in Payment Inbox as Needs Attention

A transaction with a missing reference but matching sender or amount must still remain `unknown` unless the user manually resolves it.

---

### 8.3 Duplicate Behavior

There are two duplicate cases.

Webhook duplicate:

- same signature received more than once
- ignore idempotently
- return success response

Business duplicate:

- different signature
- same expected payment reference
- expected payment already marked paid
- classify as `duplicate`
- do not update expected payment status

---

### 8.4 Overdue Behavior

Overdue is not a transaction classification.

Overdue is a display state for expected payments.

If:

- expected payment status is pending
- due date exists
- due date is in the past

Then display as:

- `overdue`

No cron is required.

No database write is required for overdue.

Overdue must never overwrite `paid`, `partial`, `overpaid`, `duplicate`, or `unknown`.

---

### 8.5 Manual Resolution Behavior

Manual resolution is only used for unknown transactions.

System must not automatically guess matches based on amount or sender.

Manual resolution must re-run the same classifier used for automatic classification.

Manual resolution must update the same persisted transaction and expected payment records used by webhook ingestion.

---

## 9. Status Model

Expected payment statuses are:

- `pending`
- `paid`
- `partial`
- `overpaid`

Transaction statuses are:

- `paid`
- `partial`
- `overpaid`
- `duplicate`
- `unknown`

`overdue` is a display-only state for expected payments.

`overdue` is never persisted as a transaction status.

`unknown` and `duplicate` are transaction statuses only.

They do not become expected payment statuses.

---

## 10. Valid Incoming USDC Transfer

A transfer is valid for Clearline ingestion only if:

- it is a token transfer
- mint equals the configured USDC mint for the active network
- recipient is the monitored wallet or monitored USDC ATA
- amount is greater than zero
- transaction signature exists

Non-USDC transfers and transfers not sent to the monitored wallet/ATA are ignored for MVP.

If a webhook payload contains no valid incoming USDC transfer, Clearline returns success and does not create a transaction record.

---

## 11. Expected Payment Status Update Rules

When a matched transaction is classified:

- `paid` transaction updates expected payment to `paid`
- `partial` transaction updates expected payment to `partial`
- `overpaid` transaction updates expected payment to `overpaid`
- `duplicate` transaction does not update expected payment status
- `unknown` transaction does not update expected payment status

If an expected payment is already `paid`, later matched transactions are classified as `duplicate`.

For MVP, Clearline does not aggregate multiple transactions into one expected payment.

The newest matched non-duplicate classification may update the expected payment status unless the expected payment is already `paid`.

---

## 12. Product Invariants

Clearline must:

- classify only incoming USDC transfers to the monitored wallet/ATA
- never guess matches by sender or amount
- match automatically only by Solana Pay reference
- allow manual resolution for unknown payments
- store valid incoming transfer payloads before classification
- preserve raw payload, signature, and extracted transfer data even when RPC fallback fails
- provide a human-readable classification reason for every stored transaction
- keep UI reads based on Supabase, not live RPC
- keep overdue as display-only
- never require the user to open an explorer to understand a classification
- never classify non-USDC transfers
- never treat overdue as an onchain event

---

## 13. Edge Cases

### 13.1 Helius Duplicate Webhook

If Helius sends the same transaction more than once:

- transaction signature must prevent duplicate inserts
- system returns success
- no reclassification occurs

---

### 13.2 Missing Reference in Enhanced Payload

If Helius enhanced payload does not expose the reference clearly:

- system runs RPC fallback
- system attempts to inspect account keys
- system compares account keys against stored expected payment references

---

### 13.3 RPC Fallback Failure

If RPC fallback fails:

- system must not crash
- transaction must remain stored if it is a valid incoming USDC transfer
- raw payload must remain stored
- signature must remain stored
- extracted transfer data must remain stored
- transaction should be classified as `unknown`
- classification reason should explain fallback failure

---

### 13.4 Transfer Without Reference

If USDC is sent directly to the monitored wallet without Solana Pay reference:

- classify as `unknown`
- show in Payment Inbox
- allow manual resolution

---

### 13.5 Already Paid Expected Payment Receives Another Transfer

If expected payment is already paid and another matched transfer arrives:

- classify new transaction as `duplicate`
- expected payment remains paid

---

### 13.6 Partial Payments Are Not Cumulative

For MVP:

- do not sum multiple partial payments
- do not support cumulative payment accounting
- do not track remaining balance
- classify each transaction independently

---

### 13.7 Irrelevant Webhook Payload

If a webhook payload contains no valid incoming USDC transfer:

- return success
- do not create a transaction record
- do not classify anything
- do not crash

---

## 14. Assumptions

- Demo runs on Solana Devnet
- Architecture is mainnet-ready by configuration
- One monitored wallet
- One USDC mint per network
- One expected payment reference per expected payment
- Supabase is the application source of truth
- Blockchain/RPC is used for ingestion and validation
- UI reads from Supabase, not directly from RPC
- Helius webhook delivery may be delayed
- Replay endpoint exists for demo insurance

---

## 15. Non-Goals

Clearline will not build:

- auth
- RLS
- multi-wallet support
- fiat payments
- Pix
- subscriptions
- invoice editor
- accounting exports
- CSV export
- email notifications
- AI features
- dashboards or charts
- custom smart contracts
- mainnet deployment requirement
- heuristic matching by amount + sender
- cumulative partial payment accounting
- production queue infrastructure

---

## 16. Demo Behavior

The demo should be understandable in under 3 minutes.

Recommended demo flow:

1. Open Payment Inbox
2. Show incoming payment states
3. Create Expected Payment A for 100 USDC
4. Generate Solana Pay link for Expected Payment A
5. Trigger exact payment for Expected Payment A
6. Show `paid`
7. Create Expected Payment B for 100 USDC
8. Trigger partial payment of 60 USDC for Expected Payment B
9. Show `partial`
10. Create Expected Payment C for 100 USDC
11. Trigger raw USDC transfer without reference
12. Show `unknown`
13. Manually resolve that unknown transfer to Expected Payment C
14. Show updated classification
15. Show proof page

Optional duplicate scenario:

16. Send another payment to Expected Payment A after it is already paid
17. Show `duplicate`

Demo must emphasize:

- Clearline is not invoicing
- Clearline classifies incoming USDC transfers
- Solana Pay references enable structured matching
- unknown transfers become exceptions
- proof page makes classification understandable

---

## 17. Ambiguities

These must not be guessed by AI agents.

If encountered, stop and ask:

- Whether to add auth or user accounts
- Whether to support multiple wallets
- Whether to support mainnet deployment
- Whether to support cumulative partial payments
- Whether to add email, CSV, or dashboards
- Whether to add heuristic matching by amount or sender
- Whether to add invoice-like features
- Whether to classify non-USDC transfers
- Whether to use live RPC reads from UI routes

Default answer for MVP:

- do not add

---

## 18. Definition of MVP Success

The MVP is successful if:

- expected payment can be created
- Solana Pay link can be generated
- devnet USDC payment can be sent
- Helius can detect the payment
- valid incoming USDC transfer is stored
- irrelevant webhook payloads are ignored safely
- reference can be extracted or fallback handled
- payment can be classified correctly
- unknown payment can be manually resolved
- proof page shows transaction details
- demo works in under 3 minutes
